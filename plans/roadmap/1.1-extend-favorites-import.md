# 1.1 Extend favorites import — remaining work

Roadmap: Phase 1.1. Forward plan for the part of import that has not shipped.

## Verified current state

`extension/src/import/favoritesImport.ts` imports three source formats today:

- **kdcro101 `.favorites.json`** — parses the JSON array; only `type === "File"`
  entries become pins; folder/group entries are logged and **skipped**
  (`favoritesImport.ts:154-160`).
- **oleg-shilo text lists** — `.vscode/fav.local.list.txt` and `.fav/local.list.txt`,
  `path` or `path|alias` per line, `#` comments.
- **howardzuo settings key** — reads `favorites.resources`.

All three dedup on resolved path via `store.addPin()` (idempotent). Detection runs
once per command invocation; there is **no standing prompt** for newly-appearing
source files. There are **no tests**.

## Remaining work

1. **Import folder/group entries into pin groups.** kdcro101 group entries are
   currently dropped. Pin groups have shipped (`ProjectPinsFile.groups`, schema v2),
   so map a source folder/group to a `PinGroup` and assign imported pins to it.
   - kdcro101 nests entries under a named group; map that name to a `PinGroup` (create
     if absent, reuse by name if present so re-import stays idempotent).
   - Malformed or unsupported entries are reported in the output channel and skipped —
     never abort the whole import (preserve the existing skip-reporting contract).
2. **Ongoing detection prompt.** When a known source file (`.favorites.json`,
   `fav.local.list.txt`, or a populated `favorites.resources`) appears or changes and
   has un-imported entries, surface a one-time, dismissible offer to import. Gate on a
   per-workspace "offered / imported / dismissed" flag in `globalState` so it never
   nags (mirror the global "offer next step, gated to once" UX rule). A
   `FileSystemWatcher` on the two filenames + a `workspace.onDidChangeConfiguration`
   listener for the settings key drive the prompt; both disposed via
   `context.subscriptions`.
3. **Project Manager + Bookmarks assessment.** Write the per-format mapping note (see
   the companion plan, `additional-import-formats.md`). If a format maps to the pin
   model without data loss it ships here; otherwise it is recorded as deferred with the
   reason. Bookmarks is line-level (it maps to line pins, which exist) — assess whether
   the mapping is lossless before committing.
4. **Tests.** Cover the mapping rules once the harness exists (Phase 4.1): kdcro101
   File-vs-folder split, group mapping, idempotency on re-run, oleg-shilo `path|alias`
   and `#`-comment handling, howardzuo relative-vs-absolute resolution, and the
   malformed-entry skip path. Until the harness lands, the import logic is structured as
   pure functions (parse → normalized entries → pins) so the tests need no VS Code host.

## Approach

- Refactor each format's parser to return a normalized intermediate
  (`{ path, label?, groupName? }[]`) before any `store.addPin()` call. This isolates
  the mapping rules as pure, testable functions and is where group support slots in.
- Group assignment reuses the existing group-creation path; do not add a parallel
  "import group" concept.
- New user-facing strings (the detection prompt, group-import summary) go in
  `src/i18n/locales/en.json` via `l10n()` — add keys as part of the change.

## Acceptance criteria

- kdcro101 folder/group entries import into matching pin groups; re-import creates no
  duplicate pins or groups.
- Malformed entries are reported and skipped without aborting the import.
- A newly-appearing source file triggers at most one import offer per workspace until
  acted on or dismissed.
- Mapping rules are documented and (once the harness exists) covered by tests.

## Dependencies

- Pin groups — shipped.
- Tests depend on the Phase 4.1 harness — see `4.1-unit-tests.md`.
- Project Manager / Bookmarks decision shared with `additional-import-formats.md`.
