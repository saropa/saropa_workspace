# 4.2 Integration smoke test

Roadmap: Phase 4.2. Depends on the harness built in 4.1.

## Verified current state

No integration harness exists (see `4.1-unit-tests.md`): no `@vscode/test-electron`, no
`runTests.js`, the `test` script targets a missing file. The unit harness (4.1) runs
pure logic without the host; the smoke test needs the **real VS Code host** to catch
manifest/activation regressions unit tests cannot.

## Remaining work

1. **Host launcher.** Add `@vscode/test-electron`'s `runTests` entry
   (`out/test/runTests.js`) that downloads/launches a VS Code host and runs the
   integration suite. Wire it as `npm test` (distinct from `npm run test:unit`).
2. **Activation assertions.** A test that:
   - activates the extension and asserts `activate()` completes without throwing;
   - asserts the view container / Pins tree view is registered;
   - asserts the core commands register (pin, run, open, configure) by checking
     `vscode.commands.getCommands(true)` contains the `saropaWorkspace.*` ids.
3. **One pin → run path end to end.** In a temp workspace folder: pin a file, assert it
   appears in the store/tree, run it, and assert a visible outcome (the run is recorded
   in telemetry / run status, or the background channel produces output). Use a trivial
   target (e.g. an `echo`/`node -e` one-liner) so the assertion is deterministic
   cross-platform.
4. **Scope the runner.** The integration suite has its own glob
   (`out/test/integration/**`) so `npm test` runs only it plus, if desired, the unit
   suite — never an accidental full sweep. Verify the reported test count matches intent.

## Approach

- Keep the integration suite minimal — it is a regression tripwire for
  manifest/activation/command-registration, not broad behavioral coverage (that is 4.1).
- Use a disposable temp folder as the workspace so the pin-and-run path does not touch
  the user's repo; clean it up in teardown.
- Cross-platform run target: prefer `node -e "process.stdout.write('ok')"` so no shell
  or interpreter assumption is baked in.

## Acceptance criteria

- The smoke test activates the extension and asserts the view container + core commands
  register.
- It exercises one pin → run path end to end against the test host with a visible,
  asserted outcome.
- `npm test` runs the scoped integration suite (count matches intent), green in CI.

## Dependencies

- **Depends on** the 4.1 harness for the mocha/compile plumbing and the
  `@vscode/test-electron` dev dependency.
