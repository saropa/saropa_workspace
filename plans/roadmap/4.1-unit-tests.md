# 4.1 Unit tests (and the harness they need)

Roadmap: Phase 4.1. The harness itself is unshipped ŌĆö building it is step one.

## Verified current state

There is **no working test harness**:

- No `extension/src/test/**`, no `*.test.ts`, no `out/test/runTests.js`.
- No `@vscode/test-electron`, no mocha config, no `pretest` script.
- `package.json`'s `test` script points at `node ./out/test/runTests.js` ŌĆö a file that
  does not exist, so `npm test` fails immediately.

So 4.1 (unit) and 4.2 (integration) both start by standing up the harness.

## Remaining work

### A. Build the harness

1. Add dev dependencies (blast-radius: new deps ŌĆö required for any test, low risk,
   standard VS Code testing stack): `mocha`, `@vscode/test-electron` (for 4.2 only),
   `@types/mocha`, and the existing TypeScript toolchain.
2. Create `extension/src/test/` with a unit suite that runs **without** the VS Code host
   ŌĆö pure-logic tests compiled and run under plain mocha (`out/test/unit/**`). A scoped
   `.mocharc` limits the default glob to the unit folder so a bare run cannot sweep
   unrelated files.
3. Wire `npm run test:unit` (mocha over the compiled unit folder) separate from
   `npm test` (which will add the 4.2 host launch). Unit tests must run in CI as a
   scoped suite with no full-host launch.

### B. Unit coverage for the load-bearing pure logic

- **Store** (`pinStore.ts`, `pinStore` helpers): load/save round-trip; auto-pin seeding
  with removal + restore (`removedAutoPins`); schema v1ŌåÆv2 migration; scope resolution
  (relative project paths vs absolute/URI global paths via `globalStoredPath` /
  `parseGlobalPath`).
- **Command builder** (`runner.ts` plan assembly + `tokens.ts`): assembling the command
  from prefix + file path + args + cwd + env; interpreter-by-extension fallback; token
  substitution (static, placeholder, interactive). These must be extracted as pure
  functions if any currently depend on the VS Code host.
- **Schedule next-occurrence** (`schedule.ts` `nextOccurrence`): next fire for `atTime`,
  `everyMs`, and the combination (earlier wins); `days[]` weekday constraint; `lastRun`
  de-duplication across a reopen; DST boundary for the time math.
- **Double-click discriminator** (`doubleClick.ts`): open-vs-run within and beyond the
  threshold window.

## Approach

- Favor testing the **pure** functions (`schedule.ts`, `doubleClick.ts`, `tokens.ts`,
  the store's path helpers) directly ŌĆö they need no host. Where command-builder logic is
  entangled with `vscode` APIs, extract the pure core first, then test it.
- Edge cases are first-class per the test rules: empty config, missing file, no
  interpreter mapping, DST boundary, reopen de-dup.

## Acceptance criteria

- `npm run test:unit` runs a scoped mocha suite with no VS Code host and passes in CI.
- Each module above has tests for its documented behavior and its known edge cases.
- The run reports the handful of intended tests ŌĆö not a swept full glob.

## Dependencies

- None blocking. This harness is a prerequisite for 4.2 and for the tests called out in
  1.1, 1.2, and the suite-integration plan.
