# 4.1 Unit tests (and the harness they need)

Roadmap: Phase 4.1. **The harness is built and a first wave of unit coverage has
shipped.** Remaining: the store IO tests that genuinely need the extension host
(deferred to 4.2).

## Shipped (2026-06-25)

**Harness** — `extension/esbuild.test.js` discovers every `src/test/**/*.test.ts`,
esbuild-bundles each to `out/test/` (the bare `vscode` import aliased to a minimal
stub at `src/test/_stub/vscode.ts`), and runs them all under Node's built-in test
runner (`node --test`). `npm run test:unit` invokes it; it exits non-zero on any
failure or when no tests are found, and runs the exact bundles it built (no glob
sweep). **This deviates from the original mocha plan below**: the project already
had an esbuild + `node:test` pattern in place (no mocha dep), so the harness
follows that — zero new dependencies, no `.mocharc`, no host launch.

**Unit coverage** (60 tests, all green via `npm run test:unit`):

- **Schedule next-occurrence** (`schedule.test.ts`): cron parse/validate, `nextCron`
  daily/weekday/step/monthly/Vixie-OR/hourly, `lastRun` same-minute de-dup,
  `nextOccurrence` earliest-of cron+interval, disabled and timing-less schedules.
- **Token substitution** (`tokens.test.ts`): `buildTokenMap` derivation + no-root
  and no-extension fallbacks; `expandTokens` longest-match, unknown-left-literal,
  distinct-unknown-once; the `SUPPORTED_TOKENS`/`buildTokenMap` drift guard.
- **Command builder** (`commandPlan.test.ts`): the pure core extracted from
  `runner.ts` into `exec/commandPlan.ts` — interpreter fallback (explicit ->
  ext-default -> shebang -> run-directly, explicit-empty-string respected),
  `isRunnablePlan`, shell quoting, and `<prefix> "<file>" <args>` assembly
  (blank-prefix and `includeFile=false` cases).
- **Store path helpers** (`pinPaths.test.ts`): the global-pin round-trip extracted
  from `pinStore.ts` into `model/pinPaths.ts` — file vs non-file scheme branching
  (`globalStoredPath` / `parseGlobalPath`), Windows-drive-is-not-a-URI, and a
  remote-URI round-trip. Verified against the `vscode.Uri` stub, so it tests the
  branch logic, not real platform fsPath normalization (that is a 4.2 concern).
- **Double-click discriminator** (`doubleClick.test.ts`): open-vs-run within and
  beyond the threshold window, plus the different-id and second-click-too-late
  cases, driven by `node:test` mock timers (Date + setTimeout) and the stub config.

The two extractions (`commandPlan.ts`, `pinPaths.ts`) are pure-logic moves; the
production callers (`runner.ts`, `pinStore.ts`) delegate to them with identical
behavior.

## Deferred to 4.2 (needs the extension host)

The store's load/save round-trip, auto-pin seeding with removal + restore, and the
schema v1->v2 migration read and write real files / `globalState` and are entangled
with the `vscode` workspace + fs APIs throughout `pinStore.ts`. Rather than stub the
whole store graph (which would test the stub, not the store), these belong in the
4.2 host suite (`@vscode/test-electron`), where they run against the real APIs.

## Approach

- Favor testing the **pure** functions directly — they need no host. Where logic is
  entangled with `vscode`, extract the pure core first (done for the command builder
  and the store path helpers), then test it against the stub.
- Edge cases are first-class per the test rules: empty config, no interpreter
  mapping, reopen de-dup, threshold boundaries. Still open: a DST-boundary schedule
  case and interactive-token resolution.

## Acceptance criteria

- `npm run test:unit` runs a scoped suite with no VS Code host and passes in CI —
  **met** (60 tests, `node --test`, no host).
- Each pure module has tests for its documented behavior and edge cases — **met for
  the pure logic**; host-dependent store IO is tracked under 4.2.
- The run reports the handful of intended tests — not a swept full glob — **met**
  (explicit bundle list, scoped discovery).

## Dependencies

- None blocking. This harness is a prerequisite for 4.2 and for the tests called out in
  1.1, 1.2, and the suite-integration plan.
