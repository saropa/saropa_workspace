# 2.1 Export / share pin sets

Roadmap: Phase 2.1 (referenced only in the competitive-gap table, where it is still
marked **"Gap"** — that marker is stale). **Shipped** — this plan records the verified
implemented state and the narrow remaining gap, not a from-scratch build.

## Verified current state

Two complementary share paths are implemented and wired:

- **Whole-set export/import to a file** — `extension/src/commands/pinSetExport.ts`.
  `exportPinSet` writes a versioned, self-describing file
  (`format: "saropa-workspace-pins"`, `version: 1`) carrying a chosen scope's pins
  **and their groups**; `importPinSet` reads and validates it, recreates groups
  (reusing a same-label group so re-import does not pile up duplicate folders), and
  adds non-duplicate pins. Idempotent: file pins de-dup on resolved path, action pins
  on label + action kind. Recipe and auto pins are excluded (they re-detect on the
  importing machine). Import never runs a pin. Reports added vs skipped.
- **Single-pin share link** — `extension/src/import/shareLink.ts` (roadmap WOW #4).
  `encodePinLink` encodes the portable subset of one pin into a clickable
  `vscode://saropa.saropa-workspace/import?data=…` URI (base64url); `decodeSharedPin`
  validates and rejects a bad/wrong-version/empty link without throwing;
  `describeSharedPin` renders the import-confirm summary. The pin-set export reuses
  `SharedPin` / `toSharedPin`, so the two paths share one portable shape.

Both are registered: `saropaWorkspace.exportPins`, `saropaWorkspace.importPins`, and
the share-link command appear in `package.json` commands + menus; l10n keys exist in
`src/i18n/locales/en.json` (`export.*`, `import.set.*`, `pinSet.scope.*`, `recipe.desc.*`).

## The gap

The implementation is complete and safe. What remains is narrow:

1. **Correct the stale roadmap marker.** The competitive-gap table still reads
   `Export / share pin sets | Gap — Phase 2.1`. It shipped — update the row to
   "Shipped (set file + single-pin link)" when ROADMAP.md is next revised.
2. **Test coverage (Phase 4.1).** No tests exist for the round-trip. Lock down:
   - export → import round-trip preserves pins, groups, and group membership;
   - idempotency: importing the same file twice adds nothing the second time (path
     de-dup for file pins, label+kind de-dup for action pins);
   - `readSetFile` rejects a non-JSON file, the wrong `format`, and an unsupported
     `version` without throwing;
   - a one-scope hand-edited file still imports (the `normalizeScope` defensive path);
   - `decodeSharedPin` rejects a malformed / wrong-version / empty link.

## Approach

- No new code surface is needed for the feature itself — it is done. The work is the
  test suite, which depends on the Phase 4.1 harness, plus a one-line roadmap edit.
- Do not add a second export format or a network share path: the no-remote-telemetry
  principle and the blast-radius gate both rule the latter out; the file + link pair
  already covers team convergence and chat-paste.

## Acceptance criteria

- Export/import round-trip and both de-dup paths are covered by passing tests.
- `readSetFile` and `decodeSharedPin` reject malformed input in tests without throwing.
- The ROADMAP competitive-gap row no longer claims this is a gap.
