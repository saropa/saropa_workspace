# 1.2 Multi-root workspace handling refinements

Roadmap: Phase 1.2. Forward plan. Much of the original scope already shipped; this
plan covers the verified gap.

## Verified current state

`extension/src/model/pinStore.ts` already handles multi-root substantially:

- Each workspace folder reads/writes its own `.vscode/saropa-workspace.json`; project
  pin paths are stored folder-relative.
- `projectPinFolder` / `projectGroupFolder` maps resolve a pin/group id back to its
  owning folder, rebuilt on every `refresh()` (`pinStore.ts:185-193`, `943-1009`).
- `vscode.workspace.onDidChangeWorkspaceFolders` is wired to `store.rescan()`
  (`extension.ts:296`), so adding/removing a folder repaints the tree and re-arms
  timers **without a reload**.
- `cwd` / `$workspaceRoot` resolve to the pin's owning folder (`runner.ts:179-190`),
  and `resolveUri` joins `folder.uri` with the relative path so the scheme is inherited
  (`pinStore.ts:269-280`).

So per-folder ownership, reactive add/remove, and cross-folder cwd/path resolution are
implemented. The remaining work is narrower than the roadmap line implies.

## Remaining work

1. **Lift the cross-folder grouping restriction.** Today a project group is created in
   the first workspace folder, and a project pin can only join a group in its own folder
   (paths are folder-relative). Decide and implement the safe cases:
   - **Tree grouping by owning folder.** When more than one folder is open, group
     project pins under their owning folder in the tree so attribution is visible. This
     is a view-layer change in `pinsTreeProvider.ts`, not a storage change.
   - **Cross-folder group membership** is the unsafe case: a single group spanning two
     folders has no single home file and breaks the folder-relative path model. Keep
     groups folder-scoped; instead allow a same-named group to appear once per folder and
     render them merged in the tree (display-only), or explicitly document that groups do
     not span folders. Pick the display-merge approach — it gives the user the "one
     group" mental model without a cross-folder storage primitive.
2. **Attribution surface.** Each project pin's tree item shows its owning folder (in the
   description, only when 2+ folders are open) so the user can tell which
   `.vscode/saropa-workspace.json` owns it.
3. **Audit the resolution edges.** Confirm with tests (Phase 4.1): a pin in folder B
   resolves `cwd` to B (not A); `$workspaceRoot` in a folder-B pin is B's root; removing
   folder A leaves B's pins and timers intact; re-adding A restores its pins from disk.

## Approach

- View-layer first: folder grouping and the attribution description live in
  `pinsTreeProvider.ts` / `pinTreeItem.ts` and read the existing `projectPinFolder` map.
  No schema change.
- Do **not** introduce a cross-folder group storage type — that violates the
  folder-relative path invariant and the blast-radius gate. Display-merge same-named
  groups instead.

## Acceptance criteria

- With 2+ folders open, project pins group under their owning folder in the tree, and
  each shows which folder owns it.
- A pin's `cwd` and `$workspaceRoot` resolve to its owning folder.
- Adding/removing a folder updates tree and timers without reload (already true — lock
  it with a test).
- Same-named groups across folders render merged in the view without a cross-folder
  storage primitive.

## Dependencies

- None blocking. Tests depend on the Phase 4.1 harness — see `../history/2026.06/2026.06.25/4.1-unit-tests.md`.
