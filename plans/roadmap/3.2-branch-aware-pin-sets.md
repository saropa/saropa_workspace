# 3.2 Branch-aware pin sets

Roadmap: Phase 3.2. **Blocked** on Multiple favorite sets (see
`multiple-favorite-sets.md`); this plan is the design to execute once that lands.

## Verified current state

- No pin-set concept exists; pins live in a single project scope + a single global
  scope (`pin.ts`, `pinStore.ts`).
- Git is read for recipe URLs only: `getCurrentBranch(folder)` builds branch web links
  (`detectors.ts:177-183`). `GitEventWatcher` (`systemEvents.ts`) watches commits/pushes
  for chain triggers but does **not** switch anything.
- So branch detection plumbing exists; branch-driven pin switching does not.

## Dependency

This feature associates a *named pin set* with a branch and switches the active set on
branch change. There is no named-set primitive yet. **Build Multiple favorite sets
first** — that delivers the `PinSet` model, the active-set state, and the switcher.
Branch-awareness is then a binding layer on top.

## Remaining work (after sets land)

1. **Branch → set association.** Per workspace, store a `branch → setId` map (in the
   project file or `globalState`, decided with the sets design). A setting
   `saropaWorkspace.branchAware.enabled` (default **off**) gates the whole feature; with
   it off, behavior is identical to today.
2. **Switch on branch change.** Watch the current branch (extend the existing git read;
   poll `HEAD` via a `FileSystemWatcher` on `.git/HEAD`, the standard branch-change
   signal). On change, if branch-awareness is on and the branch has an associated set,
   make that set active and repaint the tree.
3. **Optional on-switch pin.** Allow a designated pin to run when the set switches (e.g.
   refresh dependencies). It runs through the normal runner so the outcome is visible (no
   silent execution) — a toast names the pin and the branch that triggered it.
4. **Graceful degradation outside git.** No `.git` → the feature is inert, no errors,
   no watcher armed.

## Approach

- Reuse `getCurrentBranch` / the git-watching infrastructure already in
  `systemEvents.ts`; do not add a second git layer.
- The `.git/HEAD` watcher is disposed via `context.subscriptions`; arm it only when
  branch-awareness is enabled and a repo is present.
- All new strings via `l10n()`; the on-switch toast names the pin and branch.

## Acceptance criteria

- With branch-awareness on, the active set follows the current branch; with it off,
  unchanged behavior.
- The optional on-switch pin runs with a visible outcome.
- Outside a git repo the feature is inert and raises no errors.

## Dependencies

- **Blocks on** Multiple favorite sets — `multiple-favorite-sets.md`.
- Reuses git detection already present in `detectors.ts` / `systemEvents.ts`.
