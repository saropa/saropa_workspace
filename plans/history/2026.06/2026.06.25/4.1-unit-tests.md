# 4.1 Unit tests (and the harness they need)

Roadmap: Phase 4.1 (and the 4.2 store-IO follow-on). **The harness, the first wave
of unit coverage, AND the store-IO / prompt-memory tests are all shipped.** The
store-IO tests were originally deferred to a `@vscode/test-electron` host suite;
they instead run under the existing `node --test` harness against an fs-backed
`vscode` stub (real temp filesystem + Map-backed mementos), so no Electron download
or host launch is needed and the genuine store persistence code still executes.

## Shipped (2026-06-25)

**Harness** — `extension/esbuild.test.js` discovers every `src/test/**/*.test.ts`,
esbuild-bundles each to `out/test/` (the bare `vscode` import aliased to a minimal
stub at `src/test/_stub/vscode.ts`), and runs them all under Node's built-in test
runner (`node --test`). `npm run test:unit` invokes it; it exits non-zero on any
failure or when no tests are found, and runs the exact bundles it built (no glob
sweep). **This deviates from the original mocha plan below**: the project already
had an esbuild + `node:test` pattern in place (no mocha dep), so the harness
follows that — zero new dependencies, no `.mocharc`, no host launch.

**Unit coverage** (all green via `npm run test:unit`):

- **Schedule next-occurrence** (`schedule.test.ts`): cron parse/validate, `nextCron`
  daily/weekday/step/monthly/Vixie-OR/hourly, `lastRun` same-minute de-dup,
  `nextOccurrence` earliest-of cron+interval, disabled and timing-less schedules.
- **Schedule DST boundary** (`scheduleDst.test.ts`): in a fixed DST-observing
  timezone (set per-file; the runner isolates each test file in its own process), a
  daily 09:00 slot and a `0 9 * * *` cron stay at 09:00 wall-clock across both the
  spring-forward (23h day) and fall-back (25h day) transitions — confirming the
  local setHours/setDate math does not drift the hour a flat ms-add would.
- **Interactive run tokens** (`promptTokens.test.ts`): `hasInteractiveTokens`
  detection (and that a plain shell `${VAR}` is left alone); `cloneWithResolvedTokens`
  substitution across command/args/cwd, leaving the stored pin untouched;
  `resolveInteractiveTokens` asking once per unique token and aborting (undefined)
  on a canceled prompt — driven through settable `vscode.window` dialog handlers on
  the stub. The remembered-value defaulting (`promptMemory`) is host state and stays
  a 4.2 concern (the singleton is an inert no-op without an activated context).
- **Token substitution** (`tokens.test.ts`): `buildTokenMap` derivation + no-root
  and no-extension fallbacks; `expandTokens` longest-match, unknown-left-literal,
  distinct-unknown-once; the `SUPPORTED_TOKENS`/`buildTokenMap` drift guard.
- **Command builder** (`commandPlan.test.ts`): the pure core extracted from
  `runner.ts` into `exec/commandPlan.ts` — interpreter fallback (explicit ->
  ext-default -> shebang -> run-directly, explicit-empty-string respected),
  `isRunnablePlan`, shell quoting, and `<prefix> "<file>" <args>` assembly
  (blank-prefix and `includeFile=false` cases).
- **Store path helpers** (`pinPaths.test.ts`): the global-pin round-trip extracted
  from `pinStore.ts` into `model/pinPaths.ts` — file vs non-file scheme branching
  (`globalStoredPath` / `parseGlobalPath`), Windows-drive-is-not-a-URI, and a
  remote-URI round-trip. Verified against the `vscode.Uri` stub, so it tests the
  branch logic, not real platform fsPath normalization (that is a 4.2 concern).
- **Double-click discriminator** (`doubleClick.test.ts`): open-vs-run within and
  beyond the threshold window, plus the different-id and second-click-too-late
  cases, driven by `node:test` mock timers (Date + setTimeout) and the stub config.
- **Store IO** (`pinStore.test.ts`): the REAL `PinStore` persistence run against an
  fs-backed stub (`workspace.fs` over a temp dir, settable `workspaceFolders` /
  config, Map-backed mementos) — `ensureProjectFile` creating a current-version
  config file; a project pin round-tripping through the file across two store
  instances; `addPin` path de-dup; a global pin round-tripping through globalState;
  an auto-pin seeded from a literal pattern (the `fs.stat` branch) then removed
  (sticky via `removedAutoPins`) and restored; an auto-pin seeded from a glob
  pattern (the `findFiles` branch); the v1->current-version migration on read
  (non-destructive until a write, no pin dropped); and the synthetic "Workspace
  config" example pin. Only the host shell is faked; the store's read/write/migrate/
  seed code is the code under test.
- **Remembered run tokens** (`promptMemory.test.ts`): `promptMemory` against a fake
  `workspaceState` — remember/getValue/forget round-trip, merge-not-replace, and
  the "Run with Last Parameters" bypass (`resolveRememberedTokens`) using a
  remembered value without prompting, asking once for an unremembered token then
  remembering it, and aborting on a canceled prompt; plus `resolveInteractiveTokens`
  remembering its answer and pre-filling it on the next run.

The two extractions (`commandPlan.ts`, `pinPaths.ts`) are pure-logic moves; the
production callers (`runner.ts`, `pinStore.ts`) delegate to them with identical
behavior.

## Store IO without the host (the former "4.2 deferral")

The store's load/save round-trip, auto-pin seeding with removal + restore, and the
schema migration read and write real files / `globalState`. The earlier note said
these needed a `@vscode/test-electron` host suite because a shallow stub "would test
the stub, not the store". On re-reading `pinStore.ts` that holds only for a shallow
stub: the persistence logic (`readProjectFile` / `writeProjectFile` /
`ensureProjectFile`, the migration, `seedAutoPins`) runs entirely through
`vscode.workspace.fs.*`, `vscode.Uri`, and the context mementos. Backing those onto
a REAL temp directory + Map-backed mementos runs that genuine code; only the host
shell is faked, which is unavoidable outside Electron regardless. So `@vscode/
test-electron` was not adopted — it would have added three dependencies and a
~200 MB VS Code download for no extra fidelity over the fs-backed stub.

A genuine host suite would still add value for surface the stub cannot reach (real
command registration, tree-view rendering, multi-root path canonicalization, actual
Settings-Sync behavior). That is optional future work, not a blocker for the store
logic, which is now covered above.

## Approach

- Favor testing the **pure** functions directly — they need no host. Where logic is
  entangled with `vscode`, extract the pure core first (done for the command builder
  and the store path helpers), then test it against the stub.
- Edge cases are first-class per the test rules: empty config, no interpreter
  mapping, reopen de-dup, threshold boundaries, DST transitions, a canceled
  interactive prompt, path de-dup, sticky auto-pin removal, and a v1 migration. The
  store IO once parked for a host suite is now covered against the fs-backed stub; a
  genuine host suite is optional future work for host-only surface (see above).

## Acceptance criteria

- `npm run test:unit` runs a scoped suite with no VS Code host and passes in CI —
  **met** (`node --test`, no host).
- Each module has tests for its documented behavior and edge cases — **met**,
  including the store IO and remembered-token paths once parked for a host suite.
- The run reports the handful of intended tests — not a swept full glob — **met**
  (explicit bundle list, scoped discovery).

## Dependencies

- None blocking, and none added: the store IO runs against the existing harness with
  an fs-backed stub, so `@vscode/test-electron` was not adopted. An optional host
  suite for host-only surface (real command/tree/Settings-Sync behavior) remains
  available as future work.

## Finish Report (2026-06-25)

### Defect / objective

The store persistence layer (`pinStore.ts`) and the remembered-run-token layer
(`promptMemory.ts`) had no automated coverage. An earlier note parked these under a
prospective `@vscode/test-electron` host suite on the premise that any `vscode`
stub "would test the stub, not the store". That premise was re-examined and found to
hold only for a shallow stub.

### What changed

The store's persistence code runs entirely through `vscode.workspace.fs.*`,
`vscode.Uri`, and the context mementos. A test stub that backs those onto a real
temp directory and Map-backed mementos therefore runs the genuine
`readProjectFile` / `writeProjectFile` / `ensureProjectFile` / migration /
`seedAutoPins` code; only the host shell is faked, which is unavoidable outside the
Electron host. On that basis the host suite was not adopted — it would have added
three dependencies and a large VS Code download for no extra fidelity over the
fs-backed stub for this layer.

- `extension/src/test/_stub/vscode.ts` gained an fs-backed `workspace.fs` (over a
  real temp dir), settable `workspaceFolders` and `getConfiguration`,
  `getWorkspaceFolder`, a glob-aware `findFiles`, `RelativePattern`, `FileType`,
  `Uri.joinPath`, and a no-op `createOutputChannel`. The prior exports (the
  `EventEmitter`, the dialog handlers, `Uri.file`/`parse`) are unchanged, so the
  existing pure-logic tests that import the stub keep passing.
- `extension/src/test/_stub/context.ts` (new) provides a Map-backed
  `ExtensionContext` with `globalState` and `workspaceState` mementos, so a value
  written by one instance is read back by another sharing the same context.
- `extension/src/test/pinStore.test.ts` (new) covers config-file creation at the
  current schema version, project and global pin round-trips across store
  instances, `addPin` path de-duplication, auto-pin seeding from a literal pattern
  (the `fs.stat` branch) with sticky removal and restore, auto-pin seeding from a
  glob pattern (the `findFiles` branch), the v1-to-current-version migration on read
  (non-destructive until a write, no pin dropped), and the synthetic Workspace-config
  example pin. The version is asserted against `PROJECT_PINS_VERSION` rather than a
  hardcoded number, so a future schema bump does not break the test.
- `extension/src/test/promptMemory.test.ts` (new) covers the
  remember/getValue/forget round-trip, merge-not-replace semantics, and the "Run
  with Last Parameters" bypass (`resolveRememberedTokens`): using a remembered value
  without prompting, asking once for an unremembered token then remembering it, and
  aborting on a canceled prompt. It also confirms `resolveInteractiveTokens`
  remembers its answer and pre-fills it on the next run.
