# 3.4 Dashboard webview (Processes / Analytics / Trends)

Roadmap: Phase 3.4. **Extend existing** — a working webview already ships; this plan
turns it into the three-tab dashboard the roadmap describes.

## Verified current state

`extension/src/views/dashboardPanel.ts` is a **real, working webview**, not a stub:

- Created via `createWebviewPanel`; single-instance; opened by command
  `saropaWorkspace.openProcessMonitor` (`extension.ts:206`).
- **Strict CSP with a per-load nonce** (`dashboardPanel.ts:163-169`), no external
  script/CDN — satisfies the local-only principle.
- **Theme-aware** via `--vscode-*` variables (`dashboardPanel.ts:220-267`).
- Renders the **Processes** surface fully: sortable collapsible tool groups, live CPU
  bars, a sparkline from the heartbeat trend CSV, and confirm-gated **End task**
  (`killProcess`, refuses protected OS/Docker groups).

What is **not** there:

- It is a **single view, not tabbed**. There is no tab chrome.
- **Analytics** is a separate command rendering a virtual **markdown preview**
  (`runAnalytics.ts`, scheme `saropa-analytics`) — not in the webview.
- **Trends** exists only as the in-dashboard sparkline; there is no Trends tab over the
  scheduled reports / CSVs.

The data for all three tabs already exists: the process poll
(`processPoll.ts` — CPU delta, child rollup), the run-telemetry store
(`telemetry.ts` lifetime counts + `runStatus.ts` session outcomes), the heartbeat CSV
(`heartbeat.ts`), and the scheduled trend reports (`scheduledRecipes.ts`).

## Remaining work

1. **Tab shell.** Add a tab strip (Processes / Analytics / Trends) inside the existing
   single CSP+nonce harness — one webview, one theme binding, one piece of chrome, per
   the roadmap's "one shared webview, not three panels." Tab state is webview-local; the
   host posts the data for the active tab on demand.
2. **Processes tab.** Move the current view under the Processes tab unchanged. No
   behavior change — it already meets the acceptance criteria (two-sample CPU delta,
   child rollup, confirm-gated kill, OS/container rows not killable).
3. **Analytics tab.** Surface the run-telemetry data (`telemetry.ts` + `runStatus.ts`)
   in the webview: runs per pin, success/failure, duration trend. Keep the existing
   markdown-preview command as the **degraded fallback** when the webview is disabled —
   do not delete it.
4. **Trends tab.** Render the scheduled trend reports (test pass/fail history,
   dependency staleness, tech-debt growth) and the heartbeat CSV as charts. The markdown
   reports and CSV files remain the fallback path.
5. **Rename / generalize the entry point.** The command is currently
   `openProcessMonitor`. Add a `saropaWorkspace.openDashboard` command that opens the
   panel on a chosen tab; keep `openProcessMonitor` as an alias that opens the Processes
   tab (no breaking change). Title becomes "Saropa Dashboard."

## Approach

- All three tabs share the one webview's nonce, CSP, and `--vscode-*` bindings —
  extend the existing HTML/JS in `dashboardPanel.ts`, do not spawn new panels.
- Charts are drawn with inline `<canvas>` + the existing nonce'd script (the sparkline
  already does this); **no external chart library** — that would break CSP and trip the
  blast-radius gate (new dependency).
- Each tab degrades gracefully: Processes → existing snapshot/TreeView path; Analytics →
  the markdown preview; Trends → the markdown reports + CSV files. The feature is never
  all-or-nothing.

## Acceptance criteria

- One webview with three tabs, strict CSP + per-load nonce, no external script/network.
- Theme-aware including high-contrast.
- Processes: live two-sample CPU delta, child rollup, confirm-gated End task naming the
  exact PID, OS/container rows not killable (already met — preserve).
- Each tab degrades to its underlying data surface when the webview is off.

## Dependencies

- Process poll, run-telemetry store, heartbeat CSV, scheduled reports — all shipped.
  Processes tab can ship first; Analytics and Trends follow on the existing stores.

## Finish Report (2026-06-25)

Shipped. The single process-monitor webview is now a three-tab Saropa Dashboard
(Processes / Analytics / Trends) inside one CSP+per-load-nonce, theme-bound frame.
All five "Remaining work" items and every acceptance criterion are implemented.

### What changed

- **`extension/src/views/dashboardPanel.ts`** rebuilt as the tab host. A tab strip
  selects Processes / Analytics / Trends; tab state is webview-local and the host
  loads only the active tab's data on demand, so the live two-sample process poll
  (~1 s) runs only while Processes is showing. Message protocol added: the webview
  posts `activateTab` once its script mounts (avoiding a post-before-listener race)
  and on each tab click; the host replies with a per-tab `data` / `analytics` /
  `trends` payload, and `selectTab` lets `show()` switch an already-open panel.
  `DashboardPanel.show` now takes the `PinStore` (Analytics name resolution) and an
  optional initial tab. Every client display string is injected from the l10n
  catalog as a `STRINGS` object — no English is hardcoded in the inlined script.
- **Processes tab** carries the prior monitor unchanged: per-tool CPU bars, the
  heartbeat sparkline, sort, and a confirm-gated End task that names the PID and
  refuses OS/container groups (`isGroupKillable`).
- **Analytics tab** renders the on-device telemetry (`telemetry` lifetime counts +
  `runStatusRegistry` session outcomes), shaped host-side into display-ready text
  so the webview re-localizes nothing. The Markdown-preview command
  (`showRunAnalytics`) is preserved and surfaced as a one-click degraded fallback.
- **Trends tab** draws canvas line charts (no external library — CSP-safe): per-tool
  toolchain CPU over time from the heartbeat CSV, and a tech-debt-marker-count trend
  derived from the dated `*_debt.md` reports. Below them, every dated scheduled
  report is listed by ritual category and is clickable to open — the durable file
  fallback path.
- **`extension/src/exec/heartbeat.ts`** gained `readTrendSeries` (per-tool series)
  with its pure core split out as `parseTrendSeries(text, count)` for unit testing.
- **`extension/src/exec/trendReports.ts`** (new) lists dated reports by category,
  derives the debt-marker trend, and exposes `validateReportPath` so an `openReport`
  message can only open a real dated report inside the workspace `reports/` folder.
- **Entry point generalized**: new `saropaWorkspace.openDashboard` command (title
  "Open Saropa Dashboard", `$(dashboard)` icon, optional tab argument);
  `saropaWorkspace.openProcessMonitor` kept as an alias opening the Processes tab,
  so the heartbeat breach toast and the recipe pin keep working unchanged.

### Tests

`extension/src/test/trendSeries.test.ts` covers `parseTrendSeries`: empty/header-only
input, single-tool chronological grouping, absent-tool zero-fill on a shared x-axis,
last-`count` truncation, quoted-comma tool names, and skipping of malformed /
non-numeric rows. Full suite: 66 pass / 0 fail (`node esbuild.test.js`).

builds (`node esbuild.js`). A project-wide `tsc` reports five errors in an unrelated
