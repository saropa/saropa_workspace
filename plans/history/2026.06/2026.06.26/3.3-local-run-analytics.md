# 3.3 Local run analytics

Roadmap: Phase 3.3 (referenced only in 3.4's dependency note — there is no standalone
3.3 section). **Shipped** — this plan records the verified implemented state and the
remaining gap, which is presentation (owned by 3.4) plus tests.

## Verified current state

The data layer and an on-demand summary view both exist and are wired:

- **Telemetry store** — `extension/src/exec/telemetry.ts`. A module-level singleton
  records every pin run (manual or scheduled) into extension `globalState`: a bounded
  most-recent list (`MAX_RECENT = 20`, de-duplicated by pin id) and a per-pin lifetime
  count that survives recent-list eviction. On-device only, never transmitted. Opt-out
  via `saropaWorkspace.telemetry.enabled` (default on); when off, `record()` is a no-op
  and the Recent group is hidden. `reset()` clears it ("Reset Run History"). A one-time
  `migrateLegacy` folds the pre-telemetry bare-id recents list into the richer store.
- **Run Analytics summary** — `extension/src/commands/runAnalytics.ts`. Builds a
  read-only Markdown report from the telemetry store plus the in-memory per-session
  run-status registry: totals (pins run, total runs), most-run pins (top 10),
  the session's success/failure split with exit codes and durations, and recent runs
  with relative timestamps and a manual-vs-scheduled tag. Rendered through a virtual
  document (`saropa-analytics` scheme) so there is no temp file to clean up. Honors the
  `telemetry.enabled` toggle (a "turn it on" note when off) and reflects a reset live.
- Registered: `registerRunAnalytics` runs in `activate()` (`extension.ts:197`);
  `saropaWorkspace.showRunAnalytics` appears in `package.json` commands + menus; the
  l10n keys exist (`analytics.*`).

So "runs per pin, success/failure, duration trend, and last-run times" — the 3.4
dependency calls this "the shipped local run-telemetry store (run history + the Run
Analytics summary)" — is implemented.

2. ~~**Test coverage (Phase 4.1).** No tests exist.~~ **Done** — covered by
   `extension/src/test/telemetry.test.ts` and `extension/src/test/runAnalytics.test.ts`
   (run under `node --test` via the existing stub harness; `buildReport` was exported
   for direct assertion). The locked-down behaviors:
   - `record()` moves a re-run pin to the front, refreshes its timestamp, and increments
     the lifetime count; the recent list stays bounded at `MAX_RECENT`;
   - `enabled() === false` makes `record()` a no-op;
   - `reset()` empties recents and counts;
   - `migrateLegacy` runs at most once and preserves order, then drops the legacy key;
   - `buildReport` renders the disabled / empty / populated branches correctly, ranks
     most-run pins by lifetime count, falls back to the removed-pin marker, and shows
     the session split.

## Approach

- The feature is done; the work is tests (Phase 4.1 harness) and a cross-reference so
  3.4 reads this store rather than re-recording runs.
- Keep all reads on-device (`globalState`) — the no-remote-telemetry principle forbids
  any network path, including for a future trend view.
