# 3.2 Branch-aware pin sets

Roadmap: Phase 3.2. Multiple favorite sets has **landed** (see
`../2026.06.25/multiple-favorite-sets.md`); this plan is the design
to execute on top of it.

## Verified current state

- No pin-set concept exists; pins live in a single project scope + a single global
  scope (`pin.ts`, `pinStore.ts`).
- Git is read for recipe URLs only: `getCurrentBranch(folder)` builds branch web links
  (`detectors.ts:177-183`). `GitEventWatcher` (`systemEvents.ts`) watches commits/pushes
  for chain triggers but does **not** switch anything.
- So branch detection plumbing exists; branch-driven pin switching does not.

## Dependency

This feature associates a *named pin set* with a branch and switches the active set on
branch change. There is no named-set primitive yet. **Build Multiple favorite sets
first** — that delivers the `PinSet` model, the active-set state, and the switcher.
Branch-awareness is then a binding layer on top.

## Remaining work (after sets land)

1. **Branch → set association.** Per workspace, store a `branch → setId` map (in the
   project file or `globalState`, decided with the sets design). A setting
   `saropaWorkspace.branchAware.enabled` (default **off**) gates the whole feature; with
   it off, behavior is identical to today.
2. **Switch on branch change.** Watch the current branch (extend the existing git read;
   poll `HEAD` via a `FileSystemWatcher` on `.git/HEAD`, the standard branch-change
   signal). On change, if branch-awareness is on and the branch has an associated set,
   make that set active and repaint the tree.
3. **Optional on-switch pin.** Allow a designated pin to run when the set switches (e.g.
   refresh dependencies). It runs through the normal runner so the outcome is visible (no
   silent execution) — a toast names the pin and the branch that triggered it.
4. **Graceful degradation outside git.** No `.git` → the feature is inert, no errors,
   no watcher armed.

## Approach

- Reuse `getCurrentBranch` / the git-watching infrastructure already in
  `systemEvents.ts`; do not add a second git layer.
- The `.git/HEAD` watcher is disposed via `context.subscriptions`; arm it only when
  branch-awareness is enabled and a repo is present.
- All new strings via `l10n()`; the on-switch toast names the pin and branch.

## Acceptance criteria

- With branch-awareness on, the active set follows the current branch; with it off,
  unchanged behavior.
- The optional on-switch pin runs with a visible outcome.
- Outside a git repo the feature is inert and raises no errors.

## Dependencies

- **Built on** Multiple favorite sets (shipped) —
  `../2026.06.25/multiple-favorite-sets.md`.
- Reuses git detection already present in `detectors.ts` / `systemEvents.ts`.

## Finish Report (2026-06-26)

Status: **Shipped.** All four "remaining work" items are implemented and the
acceptance criteria are met.

### What was built

- **Branch → set binding, gated by a setting.** `saropaWorkspace.branchAware.enabled`
  (boolean, default **off**) gates the whole feature; with it off, behavior is
  identical to plain pin sets. The binding map is stored in `workspaceState` (keyed
  by branch name → `{ set, runPinId? }`). `workspaceState` was chosen over
  `globalState` deliberately: a branch→set binding is specific to one repository, and
  `globalState` is both Settings-Sync'd and shared across every workspace, so a
  binding for branch `main` would wrongly apply to every repo with a `main` branch.
  This mirrors the existing per-workspace branch-scope flag (`SHOW_ALL_BRANCHES_KEY`).
- **Switch on branch change.** A new `BranchSetBinder` (`exec/branchSets.ts`)
  subscribes to the existing `BranchTracker.onDidChangeBranch` (which already watches
  `.git/HEAD`); no second git layer was added, per the plan's approach. On a real
  branch change it activates the bound set via `PinStore.switchSet` and repaints. A
  `lastBranch` guard ensures only an actual branch change re-aligns — an unrelated
  tracker fire (folder add/remove, init) does not undo a manual set switch the user
  made while staying on the same branch.
- **Optional on-switch pin.** A binding may carry `runPinId`; after the switch the pin
  is resolved (post-switch, so a pin living in the now-active set is found) and run
  through the normal `saropaWorkspace.runPin` command, so the run is visible (no silent
  execution). A deleted pin fails safe (logged + skipped).
- **Graceful degradation outside git.** When the tracker reports no branch (no repo /
  detached-unreadable HEAD), every binder path early-returns: inert, no errors.
- **Commands + discoverability.** `Link Current Branch to Pin Set...` (pick a set, then
  an optional on-switch pin from that set's pins + globals; offers to enable the
  feature when off) and `Unlink Current Branch from Pin Set`
  (`commands/branchSetCommands.ts`), surfaced in the Pins view-title sets menu and the
  command palette. A new `PinStore.getSetPins(name)` reads an inactive set's pins
  without switching, so the link command can offer a pin from the set being linked.
